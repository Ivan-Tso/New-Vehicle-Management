# AI 保养分析模块重构完成

## 核心改进

### 1. 真正的 AI 介入
- **之前**：基于固定规则的简单判断，本质上是静态表格查询
- **现在**：调用 GLM-4.5/DeepSeek 等大语言模型，生成个性化保养建议

### 2. 手册解析与 RAG
- **之前**：上传的手册仅存储，不参与分析
- **现在**：
  - 使用 `pypdf` 自动提取 PDF 文本
  - 存储到 `uploaded_manuals.extracted_text` 字段
  - 分析时检索对应手册内容作为上下文

### 3. 多文件支持
- **之前**：仅支持单个文件上传
- **现在**：支持同时上传多个手册文件（`<input multiple>`）

### 4. 历史记录整合
- **之前**：仅基于里程的简单规则
- **现在**：结合过去 5 条维修记录，避免重复建议

## 技术实现

### 文件结构
```
vehicle-management/
├── app.py                  # 主应用，AI 分析路由重构
├── ai_maintenance.py       # 全新 AI 分析模块（RAG 模式）
├── templates/
│   └── ai_analysis.html    # 支持多文件上传和 AI 结果显示
├── .env.example            # 环境变量示例
└── AI_SETUP.md             # 详细配置指南
```

### 关键函数

#### `ai_maintenance.py`
- `extract_text_from_pdf(filepath)`: 使用 pypdf 提取文本
- `extract_text_from_file(filepath, filetype)`: 通用文本提取
- `call_llm_api(prompt, model)`: 调用 LLM API（支持 Zhipu GLM/DeepSeek）
- `analyze_maintenance_ai(vehicle, history, manual_text, lang)`: 主分析函数
- `get_manual_text_for_vehicle(vehicle_id)`: 检索车辆手册文本

#### `app.py`
- `ai_analysis()`: 重构后的路由，支持多文件上传和 AI 调用

### 数据流

```
用户上传手册 → 提取文本 → 存储到 DB
     ↓
选择车辆 → 检索手册 + 历史
     ↓
构建 Prompt（车辆信息 + 手册 + 历史）
     ↓
调用 LLM API
     ↓
生成个性化建议 → 显示
```

## 配置要求
